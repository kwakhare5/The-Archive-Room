Universal Rules — Every Project

These apply regardless of what type of project you build. Breaking any of these is why good projects still get ignored by recruiters.

RULE 01

Always live. Always.

Every project must have a working demo URL. Vercel, Netlify, HuggingFace Spaces, Railway — free options exist for every project type. A GitHub link with no demo does not exist to a recruiter.

RULE 02

Numbers, not descriptions

Every bullet on your resume needs a number. Users, requests, percentage improvement, stars, downloads, revenue — anything measurable. "Built X" → "Built X used by N people / achieving Y metric." No exceptions.

RULE 03

README is your sales page

Every repo needs: project name + one sentence, live demo link (top of README), screenshot or GIF, tech stack list, how to run locally. Repos without READMEs look abandoned. With a good README they look professional.

RULE 04

Ship fast, iterate publicly

Tweet about it on day 1. Not when it's perfect — on day 1. "Building X, here's what I have so far" gets followers and feedback. Mufeed built Devika in 20 hours and posted it. Timing and speed beat polish.

RULE 05

Know your stack decisions

For every technology you use, be able to answer: "Why did you choose this over X?" Why Next.js and not Vite? Why Supabase and not Firebase? Why Vercel AI SDK and not raw fetch? These are the exact questions every interview asks.

RULE 06

One project at a time

Never have two projects in progress simultaneously. Finish, deploy, share, get users, add to resume — THEN start the next one. Half-finished projects on GitHub are worse than no projects. Completion is the skill.

RULE 07

GitHub green every day

The contribution graph is visible on your profile. Companies open it. A graph with daily commits signals consistency and dedication — regardless of what the commits are. Even a README edit counts. One commit per day, minimum.

RULE 08

Real users over complex tech

50 people using a simple CRUD app beats a sophisticated ML system nobody uses. Companies hire for impact, not complexity. Always optimise for getting real users on your project over making the technology more interesting.
